A Soteria Primer

What is a Drill?

Backups you can prove will restore.

CURRENT SITUATION

Your team runs nightly backups. The dashboard shows green. But no one has actually tried to restore one in months. When the bad day comes — ransomware, a bad migration, a dropped table — you discover the backup is corrupted, incomplete, or takes 14 hours to restore. By then it's too late.

THE DRILL SOLUTION

A drill is a controlled restore. We pull your latest backup into a clean, isolated database, run the assertions your team defines — row counts, schema integrity, critical tables present, business invariants — and produce a cryptographically signed PDF as proof. Run on a schedule. Fail loudly when something's off. Catch the broken backup on a Tuesday, not during the outage.

WHY DRILLING

Catch silent failures

THE WHAT-MONITORING-MISSES PROBLEM

Backup software reports "success" the moment the bytes are written. It doesn't verify the bytes can be restored into a working database. Schema drift, corrupted blocks, missing WAL segments, encryption-key rotation gone wrong — all invisible until restore time.

WHAT A DRILL CATCHES

Bad dumps, truncated files, schema changes that broke a consumer, tables that quietly stopped being included, row counts that diverged from production. The thing you needed to know weeks ago.

Compliance evidence

WHAT AUDITORS ACTUALLY WANT

SOC 2, ISO 27001, HIPAA, PCI — every framework expects periodic restore testing. Auditors don't want a policy document saying you do it. They want signed, dated proof that you actually did it.

WHAT A DRILL PRODUCES

A cryptographically signed PDF for every drill — the dump fingerprint, the assertions run, pass/fail results, time taken, the signing key. Hand them the folder; the audit conversation ends in 10 minutes instead of 10 days.

Faster real recovery

THE REHEARSAL EFFECT

A drill rehearses the restore path. When the real outage hits, your team has done it a hundred times. The runbook isn't theoretical — it's the same flow they watched complete this morning.

THE OUTCOME

Restore time goes from "we'll have to figure it out" to a number you've measured every day for the past month. RTO stops being aspirational.

THE FIVE STEPS OF A DRILL
1
Fetch the dump

Soteria pulls the latest backup from your storage — S3, GCS, Azure Blob, a local file path. Read-only; we never touch the source database.

2
Restore into a clean target

An isolated, ephemeral database — a fresh Postgres instance, spun up just for this drill, torn down when it's done. Nothing touches production.

3
Run your assertions

The checks you defined — "users table has at least 10,000 rows," "no NULLs in payments.amount," "schema matches this fingerprint," "last order is < 26 hours old." Plain SQL.

4
Sign the evidence

A PDF report — dump fingerprint, each assertion, pass/fail, timings — signed with an Ed25519 key. Stored encrypted. Anyone can verify it later with the public key.

5
Tear down and alert

The target database is destroyed. If anything failed, you hear about it immediately — email, Slack, webhook. If everything passed, the green PDF lands in your evidence vault.

WHAT'S INCLUDED
Connector library
Soteria reads dumps from S3, GCS, Azure Blob, Cloudflare R2, or any local/SFTP path. Postgres first, MySQL and SQL Server next. Add your own format with a small plugin.
Isolated restore runner
Every drill restores into a fresh, ephemeral target — a one-off Postgres instance on Fly Machines or a sandbox of your choosing — torn down the moment the drill ends. Zero touch on production.
Assertion engine
Write your business invariants in plain SQL — row counts, freshness checks, foreign-key integrity, anything queryable. Soteria runs them, captures the results, and fails the drill on any red.
Evidence vault
Every drill produces a signed PDF — dump fingerprint, assertions, pass/fail, timings — Ed25519-signed and AES-GCM encrypted at rest. Verifiable by anyone with the public key, years later.
Scheduling
Cron-style cadences per database, plus on-demand drills via UI or API. Pause for maintenance windows. Different SLAs for different tiers of data.
Failure alerts
When a drill fails, your team hears about it — email, Slack, generic webhook — with the exact failing assertion and a link to the evidence PDF. Pages only on real signal.
Audit-ready reports
A per-database history page: every drill, every assertion result, pass-rate trend lines, RTO over time. Export the evidence pack as a zip for an auditor or a board meeting.
Tamper-evident chain
Every drill links to the prior one by hash. Deleting a failure leaves a visible gap. The chain is verifiable end-to-end; auditors love it.
Read-only on production
Soteria never writes to your source database and never needs production credentials beyond the read access to the dump location. The drill happens entirely in the sandbox.
